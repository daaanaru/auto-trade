## 0. エグゼクティブ・サマリー

ペーパートレード卒業判定3択のうち **A案（vol_div単独条件付き卒業 + 実弾投入）** を実行する。
現行 `live_trade.py` はDRY_RUN骨格しかないため、**4レイヤー構成**に再設計する。

**設計原則**: 「**決定論はプログラム、判断はLLM**」を厳守。

| レイヤー | 担当 | 役割 |
|---|---|---|
| **L1 シグナル** | プログラム | vol_divのMFI divergence判定（現状維持） |
| **L2 サイジング・リスク** | プログラム | ATR SL/TP、日次損失ブレーカー、連敗カウンタ、サイズ計算 |
| **L3 発注実行** | プログラム | ccxt bitFlyer、冪等性、リトライ、約定記録 |
| **L4 LLM判断ゲート** | LLM（Claude Max） | **日次Go/No-Go + 異常事態レビュー**のみ。個別エントリー検証はやらない |

**ペーパー期の失敗を反省した3原則**:

1. **LLMを毎トレードに入れるな** — 27日間のfail-closedが127回の取引を激減させた真因
2. **1銘柄サイズを倍以上に** — 購買力活用率23.7%はサイジング設計ミス
3. **過去データへの係数調整で卒業を偽装するな** — 戦略はvol_divだけ、パラメータは凍結

## 2. 設計原則

### 2.1 プログラムvsLLM分離マトリクス

| 判断種別 | 担当 | 理由 |
|---|---|---|
| テクニカル指標計算（MFI/EMA/VO） | **プログラム** | 決定論・高速・無料 |
| ダイバージェンス検出 | **プログラム** | 算術判定 |
| シグナル生成（BUY/SELL/HOLD） | **プログラム** | vol_div戦略内で完結 |
| ポジションサイズ計算 | **プログラム** | ATRベースのリスク計算 |
| SL/TP価格決定 | **プログラム** | ATR倍率の機械計算 |
| 発注/約定/冪等性管理 | **プログラム** | DBトランザクション的 |
| ブレーカー発動（損失・連敗・乖離） | **プログラム** | 機械的しきい値 |
| ---- | ---- | ---- |
| **日次Go/No-Go判断**（重大ニュース解釈） | **LLM** | 文脈解釈が必要 |
| **異常事態の事後レビュー**（なぜ負けたか） | **LLM** | 非構造情報の要約 |
| **週次戦略死活判定**（続行/停止/調整） | **LLM** | 複数情報源の統合 |
| **卒業判定レビュー**（ペーパー→実弾の移行可否） | **LLM** | 定性的総合判断 |

### 2.2 フェイルセーフ原則

- **LLM沈黙時のデフォルト**: 「Go/No-Go判断」は **前日の判断を継続**（fail-same）。個別エントリーをブロックしない
- **プログラム例外時のデフォルト**: 全ポジション強制決済 + Discord緊急通知 + launchd停止
- **取引所API異常時のデフォルト**: 発注を見送る（既存ポジションは維持）
- **冪等キー**: 各シグナル判定サイクルに`session_id`を付与し、二重発注を防ぐ

### 2.3 データ駆動原則

全ての判断・発注・ブレーカー発動は **構造化JSONログ** に残す。LLM層の判断もログに含める。後日、LLMの判断をバックテストに使える形にする。

## 4. 各レイヤー詳細設計

### 4.1 L1 シグナル層（プログラム）

**ファイル**: `strategies/volume_divergence.py`（既存・変更なし）

**凍結事項**:
- パラメータは`optimized_params.json`の`vol_div`キーを凍結
- ペーパー期に使った設定をそのまま実弾に持ち込む
- **係数いじりは禁止**（過学習防止）

**入力**: CCXTFetcher経由の1h足 120本（vol_divは日足で検証してきたが、サイジングで回転を増やすため **1h足に昇格**する。※1h昇格についてはStage 1で少量実弾前にバックテスト再検証が必要。Stage 0では日足維持）

**出力**: `signal ∈ {-1, 0, 1}`

**ガード**:
- EMA200がNaNならスキップ（最低200本のデータ必須）
- スイングロー/ハイが2つ未満ならHOLD
- `div_window + swing_lookback` 未満のデータ長ではHOLD

### 4.2 L2 リスク管理層（プログラム・**新規作成**）

**ファイル**: `risk_manager.py`（新規）

```python
# 擬似コード
class RiskManager:
    def __init__(self, config: dict):
        self.stage = config["stage"]  # 0/1/2/3
        self.state_file = config["state_file"]
        self.breaker_file = config["breaker_file"]

    # === サイジング ===
    def position_size(
        self,
        capital_jpy: float,
        atr: float,
        price: float,
        signal: int,
    ) -> dict:
        """
        ATRベースのサイジング。
        - risk_per_trade: 資本の1%（Stage 1）、2%（Stage 2）、3%（Stage 3）
        - stop_distance: ATR × 1.5
        - position_size = (capital * risk) / stop_distance
        - max_position_jpy: Stage毎の絶対上限(Stage1=3000円, Stage2=30000円, Stage3=150000円)
        """
        risk_pct = {0: 0.01, 1: 0.01, 2: 0.02, 3: 0.03}[self.stage]
        stop_distance = atr * 1.5
        risk_amount = capital_jpy * risk_pct
        if stop_distance <= 0:
            return {"size": 0, "reason": "ATR不正"}
        size = risk_amount / stop_distance
        order_jpy = size * price
        max_jpy = self._stage_max_order()
        if order_jpy > max_jpy:
            size = max_jpy / price
        return {"size": size, "stop_distance": stop_distance, "risk_amount": risk_amount}

    # === ブレーカー ===
    def check_breakers(self, state: dict) -> Optional[str]:
        """発動していれば理由文字列、無事ならNone"""
        # 月次-10%
        if state["monthly_pnl_jpy"] / state["config"]["initial_capital_jpy"] <= -0.10:
            return "MONTHLY_LOSS_10PCT"
        # 日次-3%
        if state["daily_pnl_jpy"] / state["config"]["initial_capital_jpy"] <= -0.03:
            return "DAILY_LOSS_3PCT"
        # 連敗3回
        if state["consecutive_losses"] >= 3:
            return "LOSING_STREAK_3"
        # ペーパー乖離（週次チェックで更新される指標）
        if state.get("paper_deviation_pct", 0) >= 30:
            return "PAPER_DEVIATION_30PCT"
        return None

    # === SL/TPチェック（15分毎monitorで呼ぶ）===
    def check_exit(self, position: dict, current_price: float) -> Optional[str]:
        entry = position["entry_price"]
        side = position["side"]  # "long"/"short"
        atr = position["atr_at_entry"]

        sl_distance = atr * 1.5
        tp_distance = atr * 3.0

        if side == "long":
            if current_price <= entry - sl_distance:
                return "STOP_LOSS"
            if current_price >= entry + tp_distance:
                return "TAKE_PROFIT"
        else:
            if current_price >= entry + sl_distance:
                return "STOP_LOSS"
            if current_price <= entry - tp_distance:
                return "TAKE_PROFIT"

        # トレーリング発動（+1.5ATR到達後）
        if position.get("trailing_active"):
            hwm = position["high_water_mark"]
            if side == "long" and current_price <= hwm - atr * 0.75:
                return "TRAILING_STOP"
            if side == "short" and current_price >= hwm + atr * 0.75:
                return "TRAILING_STOP"

        return None
```

**ブレーカー一覧**:

| ブレーカー | しきい値 | 発動時動作 |
|---|---|---|
| 日次損失 | -3% | 当日のエントリー禁止、既存ポジは維持 |
| 月次損失 | -10% | 全ポジ強制決済、翌月1日まで停止 |
| 連敗 | 3連敗 | 48時間クールダウン |
| ペーパー乖離 | 30% | 全ポジ強制決済、週次レビュー待ち |
| ATR取得失敗 | - | 当該シグナルを見送り |
| API異常 | - | エントリー見送り、既存維持 |
| 価格スパイク | ±5%/1分 | 当該シグナルを見送り |

### 4.3 L3 発注実行層（プログラム・**live_trade.py刷新**）

**ファイル**: `live_trade.py`（既存を改修）

**新規機能**:

1. **冪等性キー**:
   - 各シグナル判定サイクルに`session_id = hash(timestamp_hour + signal)`を付与
   - `live_trade_idempotency.json`に記録済みIDは再発注しない
   - launchd二重起動による二重発注を防ぐ

2. **リトライ機構**:
   - API異常時は3回まで指数バックオフ（1s → 2s → 4s）
   - 最終失敗はDiscord通知＋halt_flag立ち上げ

3. **約定確認**:
   - 発注後、`fetch_order(order_id)`で約定価格を再取得
   - 約定価格が参考価格から ±2% 以上乖離したらHALT

4. **DRY_RUN → LIVE遷移ゲート**:
   - `.env`の`LIVE_TRADE_STAGE=0/1/2/3`で段階制御
   - Stage 0: DRY_RUNのみ（コード実行は全フロー、発注だけシミュ）
   - Stage 1以上: `.env`にAPI鍵が無ければ自動的にStage 0に降格

5. **発注前必須チェックリスト**:
   - [ ] ブレーカー未発動
   - [ ] halt_flag未立ち上げ
   - [ ] 冪等性キー未使用
   - [ ] 取引所残高 > 注文金額 × 1.05（手数料マージン）
   - [ ] ポジションサイズ > 取引所の最小単位
   - [ ] 1/5以上のチェック失敗 → 発注見送り + Discord警告

### 4.4 L4 LLM判断層（**限定的使用**）

#### 4.4.1 日次Go/No-Go判断（毎朝8:55 JST）

**ファイル**: `daily_go_nogo.py`（新規）

**目的**: 重大ニュース・相場環境の急変を踏まえ、「本日エントリー可」を1回判定する。

**呼び出し**:
```
claude -p "{prompt}" --output-format json
```

**プロンプト入力**:
- 前日のvol_divシグナル履歴
- 前日のポジション/PnL
- ペーパー乖離率
- 市場レジーム（regime_detector.py出力）
- ブックマーク巡回の最新ニュース10件（`80_ナレッジ/bookmarks/`から）

**出力スキーマ（厳格）**:
```json
{
  "verdict": "GO" | "NO_GO",
  "confidence": 0.0-1.0,
  "reason": "1〜3文の日本語",
  "alerts": ["注意点があれば列挙"]
}
```

**フェイルセーフ**:
- **LLM沈黙/エラー → verdict=GO（前日継続）** ← 旧validator反省点
- パース失敗でもエントリーをブロックしない
- ただし連続3日エラーなら halt_flag 立ち上げ + 幕府通知

**結果の使われ方**:
- `verdict=NO_GO` なら当日のエントリーを全拒否
- 既存ポジションのSL/TPは影響を受けない
- 判断結果を `daily_go_nogo_log.json` に蓄積

#### 4.4.2 異常事態の事後レビュー（ブレーカー発動時・即時）

**ファイル**: `incident_reviewer.py`（新規）

**トリガー**: ブレーカー発動・強制決済・連敗閾値超過

**動作**:
- 直近7日の取引ログ + 市場データをClaudeに渡し、原因分析を受け取る
- 結果を `incidents/YYYYMMDD_HHMM.md` に保存
- Discord通知に要約を同梱

**LLMの役割**: **判断ではなく、要約と仮説提示**。運用停止/継続の決定権は持たない。

#### 4.4.3 週次戦略レビュー（日曜22:00 JST）

**ファイル**: `weekly_strategy_review.py`（新規）

**入力**: 過去7日の全トレード + ペーパーとの乖離 + 市場レジーム変化

**出力**:
- 今週の勝因/敗因（要約）
- 次週の調整提案（パラメータ、ポジションサイズ、Stage昇格可否）
- **ただし実行権はなし**。提案を `weekly_reviews/YYYY-MM-DD.md` に残すのみ

**上様の御裁可**:
- 月曜朝、上様が週次レビューを読み、**Stage昇格/パラメータ変更を承認するか決める**
- 承認なしでは変更は一切起こらない

## 13. 本設計が「プログラム vs LLM分離原則」を守れている証拠

| 判断 | 担当 | 理由 |
|---|---|---|
| 「今vol_divのBUYシグナルが出ているか」 | プログラム | MFI/EMA/VOの算術 |
| 「このBUYシグナルに乗るか」 | **プログラム（ブレーカー+リスクマネ）** | ←ペーパー期の反省（毎トレードLLM禁止） |
| 「いくらで買うか」 | プログラム | ATR × リスク% の公式 |
| 「本日そもそも取引するか」 | **LLM（1日1回）** | ニュース文脈解釈が必要 |
| 「損切り・利確の執行」 | プログラム | 機械判定 |
| 「なぜこの週は負けたか」 | **LLM（週1回）** | 要約・仮説 |
| 「Stage昇格するか」 | **上様（LLMは提案のみ）** | 最終意思決定権は人間 |

LLM呼び出し回数の見積:
- 日次Go/No-Go: 1回/日 × 30日 = **30回/月**
- 週次レビュー: 1回/週 × 4週 = **4回/月**
- 事後レビュー: 月2〜3回想定 = **3回/月**
- **合計: 約37回/月**

※Claude Max契約内で賄えるため **追加費用ゼロ**。
※ペーパー期のentry_validator（取引毎）は1日3〜20回想定だったため、**1/10以下に削減**。

### 16.3 重大な発見

#### 発見①: 新設計は歴史データで機能する ✅

- Stage 1〜3 全て黒字、Sharpe 0.34〜0.40、DD -3.8〜-10.3%
- **ペーパー期の卒業条件（勝率≥40%、Sharpe≥0.5、DD≤-15%）を勝率・DDは余裕でクリア**
- Sharpe 0.5基準だけは微妙（0.34〜0.40）。ただし取引数16/2年は極少サンプルであり参考値

#### 発見②: 「25万円現金アイドル問題」を歴史が証明 ✅

**同一シグナル・同一期間**で比較:
- 旧設計（Paper_Old、¥300K・4%・上限¥12K）: **+0.31%**
- 新設計（Stage1、¥10K・1%・上限¥3K）: **+3.41%**

**取引数・勝率は完全に同じ**にもかかわらず、**リターンが11倍の差**。これは設計書§1.1および上様の指摘「購買力活用率23.7%は機会損失」を歴史で証明した結果にござる。

元凶は旧設計の **1銘柄上限¥12,000（4%）** が、本来¥12,000リスクを取れる状況でも 極小サイズしか張れない構造にあった。新ATRサイジングでは **資本の1%をATR×1.5で割ってサイズ決定** するため、ボラに応じて適正サイズが出る。

#### 発見③ 🚨: Stage 3のブレーカー設計に論理バグ

Stage 3では **日次ブレーカーが2年で5回発動** した。原因は明白:

| 項目 | 値 |
|---|---|
| 1取引リスク | 資本の3% = ¥6,000 |
| 日次損失限度 | 資本の3% = ¥6,000 |
| **1敗 ≈ 日次限度到達** | **即ブレーカー発動** |

発動履歴は **全てSTOP_LOSSヒット直後の同日内** であり、「1敗=1ブレーカー」という病的結合が発生している。日次-3%ブレーカーの目的は「連続被弾を止める」ことであり、**1敗で発動するのは設計意図に反する**。

**影響**: 翌日のエントリー機会を1回失う。発見①で見たように Stage 3でもリターンは+7.8%に届いたが、**これは設計が救った成果ではなく、単に残り期間で取り戻しただけ**。放置すれば連鎖的に機会損失が増える。

### 16.4 設計修正案（§6ガードレール表の修正）

```diff
| **損失** | 日次-3% | 当日新規停止 |
- Stage 3（リスク3%）では1敗で即発動する論理バグ
+ 修正A: 日次損失ブレーカーを「資本の risk_pct × 2 + 0.5%」の動的しきい値にする
+        Stage 1: -2.5%, Stage 2: -4.5%, Stage 3: -6.5%
+ または
+ 修正B: Stage 3のrisk_pctを3% → 2%に引き下げ、日次-3%を維持
```

**将軍の推奨: 修正A（動的しきい値）**
理由: Stage毎にrisk_pctが変わる設計なので、ブレーカーも連動させる方が整合的。

### 16.5 未検証・要注意事項

- **LLM Go/No-Go層は未検証** — 「常にGO」を仮定した上限値。LLMが止めたらリターン低下の可能性
- **vol_div取引頻度が極少** — 2年で43シグナル、年22取引。Stage 0（1週間）では1〜2シグナルしか出ない可能性高く、Stage 0合格条件「5件以上」は日足では過大要求
- **BTC-JPYのyfinance精度** — BTC/USDTの円換算の可能性があり、bitFlyer実勢との差分は未検証
- **2024-04-11〜2026-04-11の相場環境** — BTC上昇＋下落の両方を含むが、極端なレジーム（例: 2022年クラッシュ）は未カバー
- **1h足での再検証が必要** — 現設計は1h昇格を想定。取引頻度が数倍に増える可能性あり、リスク管理の再検証が要る

## 19. 本検証で判明した構造的洞察

### 19.1 「時間軸劣化パターン」

全戦略に共通する現象: **時間軸を下げるほど勝率が monotonic に下がる**

| 戦略 | 日足勝率 | 1h勝率 | 15m勝率 |
|---|---|---|---|
| vol_div | 68% | 47% | 30% |
| volscale | 44% | 46% | 27% |
| monthly | 55% | 44% | 29% |
| bb_rsi | 50% | 43% | 20% |
| sma | 54% | 51% | 34% |
| rsi_rev | 55% | 41% | 23% |

**この現象の意味**:
1. 日足の"高勝率"はサンプル小による偶然（ノイズ）
2. 時間軸を下げてサンプルを増やすと真の姿（ランダムウォーク）が見える
3. テクニカル指標は手数料・スリッページを超えるエッジを持たない

### 19.2 統計的有意性の罠

- N=16〜74程度の取引で「勝率60%！」と喜んでいたが、Wilson 95%CI は 40〜80% 程度
- **真の勝率は50%前後の可能性が常にあった**
- ペーパー期の実績（勝率52%, 127取引）は、この「真の50%」に収束する過程だった可能性

### 19.3 設計のbreaker/cooldownバグ

§18検証中にも発見（詳細は §16.3 以降参照）:
- 日足におけるcooldown=48バー=48日が過剰
- Stage毎のrisk_pctとdaily_loss_limit_pctの非対称で「大きく張る方が停止機能が効かない」バグ

この2点は「戦略がない」問題とは別に、設計側の欠陥として記録する。
